## What Went Well

**1. Anti-scope-creep discipline**
Every story carried an explicit "What this story does NOT include" section. Dev agents enforced it consistently. Story 1.1 did not scaffold CDK stacks. Story 1.3 did not implement auth middleware. The scaffolding never leaked forward.

**2. Architecture boundary enforcement**
`api/src/secrets/ssm.ts` as the only SSM surface, `api/src/ddb/*` reserved for Story 2.3, Zod-as-single-source-of-truth — these held across all six stories. No boundary violations found at review.

**3. Behavioral test coverage**
Test count grew from 5 to 200+ with no snapshot tests. The argon2id timing-uniformity tests (Story 1.4), the blackout-check timezone edge cases (Story 1.6), and the PerformanceModeContext provider tests (Story 1.5) are examples of tests that encode genuine behavioral intent.

**4. Code review step caught real bugs in every story**
Notable catches: `shouldRedirectOn401` was dead code until 1.4 review; `s3:*` and `lambda:*` were over-broad on the deploy role (1.3 review); `PerformanceModeProvider` only wrapped the `ready=true` branch (1.5 review). The review step is load-bearing — not optional.

**5. Deferred-work tracking worked**
The `deferred-work.md` + in-file back-reference pattern created accountability. Three items were opened and explicitly resolved across stories, with citations.

**6. Progressive story composition**
Each story's "previous story intelligence" section accurately transferred context. No story had to backtrack on predecessor decisions.

**7. Codebase feels solid**
Sandy's assessment: the code is clean enough to build on. Architecture is sound. No structural concerns entering Epic 2.

## Challenges and Lessons Learned

**1. "Deferred to Sandy" is not a reliable handoff mechanism**

The single biggest process failure of Epic 1. Three stories (1.3, 1.4, 1.6) ended with manual AWS deployment steps described in prose inside the Dev Agent Record. The stories were marked done. Sandy missed all three. The live site has never been deployed.

**Root cause:** Human-required steps written as narrative prose in the Dev Agent Record, after hundreds of lines of implementation notes, with no structural enforcement mechanism. Once the story was marked done in sprint-status.yaml, there was no signal that Sandy still had action items outstanding.

**Fix for Epic 2:** Any human-required verification step (AWS deploy, live smoke test, GitHub config) must be an unchecked task checkbox in the story spec — explicitly the last checkbox before the story closes. If the box is unchecked, the story is not done.

**2. Review finding volume in Story 1.3 (22 items)**

The infrastructure story had 16 patch findings and 6 deferred items. Some were consequential: `cloudformation:DescribeStacks` missing from the bootstrap bucket (would have broken the first deploy), WAF returning 403 instead of 429, route53 record names hardcoded instead of derived from props.

**Lesson:** CDK infrastructure stories carry higher review-finding risk than application stories. The full operational context (IAM trust boundaries, CloudFormation execution permissions, WAF behavior) is hard to verify from code alone. Budget more review attention for infra stories.

**3. Biome and tsconfig coverage discovered late**

Story 1.6's verification pass had to add `infra/scripts/**` to both `infra/tsconfig.json` and `biome.json`. The scripts folder was created during the story — coverage was not added at creation time. Same pattern with `web/test-output/` in Story 1.2.

**Fix:** When a new directory is created, add it to Biome and tsconfig includes in the same commit.

**4. AC-7 seam was dead code in Story 1.4 until review caught it**

`shouldRedirectOn401` was correctly implemented and tested in isolation, but `RequireAuth` in `router.tsx` called `<Navigate to="/login">` unconditionally, bypassing the function entirely. The function existed; the call site didn't use it. The review caught this.

**Lesson:** End-to-end behavioral paths are harder to verify than unit tests. The call graph from UI interaction → decision function needs explicit integration test coverage, not just unit tests on the function.

## Key Insights

1. The story spec format works well for dev agent implementation — the combination of ACs, tasks, dev notes, and anti-scope-creep sections produced consistently correct code.
2. The review step is essential and should remain a hard gate per story.
3. Human action items need structural enforcement, not prose documentation.
4. The architecture's boundary rules are valuable and worth the discipline cost — they held across six stories.
