## What Went Well

This is the first epic transition where parallel-bucket items moved — the Epic 1→2 gap closed.

**2. Story 3.5 paste-to-parse landed clean using a new pattern**

The highest-risk story in the epic. The team executed it as a design-note-first delivery: `paste-to-parse-design.md` was approved 2026-06-19, then Story 3.5's spec referenced it with `design_ref:` frontmatter and quoted the algorithm verbatim. Parser + normalizer + matcher + UI all landed in one pass, 79 new tests, JW implemented inline (~30 lines), all five worked AC-4 scores verified, NFR-3 500ms budget met with the 20-row × 100-song benchmark.

**3. `compareLww<T>` reuse worked as forecast from Epic 2**

Story 2.3 retro point 6 documented that `compareLww` was deliberately made generic so Story 3.1 could reuse it unchanged. Story 3.1's PUT route imports `compareLww` from `../lww.js` directly — no per-record-type override, no duplication. The forward-looking design from Epic 2 paid for itself in Epic 3.

**4. Hook-only consumption contract (AR-45) held across all 6 stories**

Every UI surface — `home.tsx`, `setlist-overview.tsx`, `setlist-creation.tsx`, `parse-row-status.tsx` — imports `useSetlists` / `useSetlist` / `useSetlistMutation` / `useSongMutation` / `useSongs`. Zero direct imports of `outbox.ts` or `flusher.ts` from UI code. Drag-reorder in 3.6 fires whole-record PUTs on every drop through `useSetlistMutation.saveSetlist`. The boundary held under the most write-heavy interaction in the project so far.

**5. Whole-record PUT semantics (AR-23) applied consistently**

Every mutation handler (`handleRename`, `handleAnnotationChange`, `handleReorder`, Save in creation) builds the FULL `SetlistPutInput` by `structuredClone(setlist.sections)` (3.6) or `.map()` chains (3.3, 3.4), only changing the relevant field. `SetlistPutInputSchema.strict()` would have caught any missing field; it didn't fire. Story 3.3's Dev Notes flagged this as Lesson #3 and 3.4–3.6 reapplied it.

**6. Native HTML5 Drag and Drop (Story 3.6) — zero new deps, full functionality**

AR-46 bars new dependencies. The team explicitly considered and rejected `react-dnd`, `dnd-kit`, `@hello-pangea/dnd`, `react-beautiful-dnd`. Native DnD plus a midpoint heuristic for above/below drop position works on the single MacBook surface, including cross-section drag, and synthesises clean test events via a `fireDragEventAt` helper that documents jsdom's DragEvent `clientY` quirk inline.

**7. Test growth was proportional and behavioral, not snapshot theater**

3.1 +57 (api 30 + web 27), 3.2 +40, 3.3 +39, 3.4 +21, 3.5 +79, 3.6 +20. Story 3.5's matcher test exercises the JW scores from AC-4's worked table; the benchmark verifies NFR-3. Story 3.6's drag tests synthesize `DragEvent`s with clientY patched in. No snapshot tests anywhere in the epic.

**8. Design-note-first is a new transferable pattern**

For Story 3.5, Sandy approved `paste-to-parse-design.md` as its own committed artifact BEFORE the story spec was written. The story spec then carried `design_ref:` in frontmatter and quoted "Implementation conforms to paste-to-parse-design.md (approved 2026-06-19)". This separates "what the algorithm IS" (design note, approved once) from "what the AC checks" (story spec, dev-facing). Candidate for Epic 4's Performance Mode entry behavior.

## Challenges and Lessons Learned

**1. The "marked-done-but-uncommitted" Epic 2 gap mutated into a "bundled-with-next-story" gap**

Epic 2 retro's #1 lesson was: stories marked done with uncommitted changes (twice in one epic). The fix was a "Task N — Commit checkpoint" added to every story spec.

The fix was applied — all six 3.x story specs have a Commit checkpoint task. But every one is **deferred** with boilerplate "_orchestration workflow owns the commit after review steps pass_" and remains unchecked. In practice:

| Story | Spec status | Actual commit | Result |
|---|---|---|---|
| 3.1 | Task 8 deferred, unchecked | ee2f0ff "Implement story 3.1" | Clean — own commit |
| 3.2 | Task 8 deferred, unchecked | 68e45ba "Implement story 3.2" | Clean — own commit |
| **3.3** | **Task 6 deferred, unchecked** | **bundled into 94dfdff "Implement story 3.4"** | **No own commit message** |
| 3.4 | Task 6 deferred, unchecked | 94dfdff "Implement story 3.4" (carries 3.3's files too) | Commit conflates two stories |
| **3.5** | **Task 8 deferred, unchecked** | **bundled into b16d2dd "Implement story 3.6"** | **No own commit message** |
| 3.6 | Task 9 deferred, unchecked | b16d2dd "Implement story 3.6" (carries 3.5's files too) | Commit conflates two stories |

`git diff-tree b16d2dd` shows `parse-row-status.tsx`, `paste-parse/matcher.ts`, `paste-parse/parser.ts`, `paste-parse/normalize.ts` — all Story 3.5 files — committed alongside Story 3.6's drag-reorder changes under a commit titled "Implement story 3.6". Same pattern for 3.3 inside 94dfdff.

**Root cause:** The "Commit checkpoint" task is in the story spec but **explicitly deferred to orchestration** with no actual per-story granularity contract. The epic-run workflow does the commit, and when stories 3.4 and 3.6 ran with uncommitted 3.3 / 3.5 files in the tree, it bundled them.

**Impact:** Archaeology loss. Six months from now, `git log` for "when did paste-to-parse land?" returns "Implement story 3.6". A bisect to find a paste-to-parse regression would point at the drag-reorder commit.

**Fix for Epic 4:** Either (a) drop the `_deferred_` boilerplate from new story specs and require the dev-story to actually commit, OR (b) update the epic-run skill (`.claude/skills/epic-run/`) to commit between dev-story and the next story's spec creation, with the commit message matching the story key. **(b) is the lower-friction fix** and matches Sandy's preference for orchestrated cadence.

**2. Story file `Status: review` is divergent from sprint-status `done` — six times in one epic**

Every 3.x story file header says `Status: review`. `sprint-status.yaml` (in the uncommitted diff Sandy is about to commit) marks all six as `done`. The story file `Status` field is driven by the dev-story workflow ("→ review"), and sprint-status is driven independently by the user. The two have diverged.

Not load-bearing today — `sprint-status.yaml` is the canonical source — but it makes the story files misleading as standalone artifacts and creates an ambiguity for next-story-creation skills about which baseline to trust.

**Fix for Epic 4:** Either (a) update the dev-story workflow to flip the story file's `Status:` to `done` when sprint-status flips, OR (b) document in CLAUDE.md that `sprint-status.yaml` is the only authoritative status and story-file headers are advisory.

**3. `readAtmosphere()` is now copied verbatim in 4+ files**

Story 2.6 introduced `readAtmosphere()` in `song-detail.tsx`. Stories 3.3, 3.4, 3.5 each copied it verbatim into their new components per Dev Notes guidance ("copy — do NOT import"). The copy count is now `song-detail.tsx`, `section-heading.tsx`, `setlist-song-row.tsx`, and at least `setlist-creation.tsx`.

The Dev Notes rationale for copying was "each consumer has its own copy" but never said *why*. The function is 4 lines, pure, no React deps, no global state. Three copies is fine; five copies starts to smell. If Sandy ever changes atmosphere detection (e.g., a third atmosphere, or move to context), there are 5 places to update with regression risk if one is missed.

**Fix for Epic 4:** Either extract `readAtmosphere()` to `web/src/lib/atmosphere.ts` (one-liner module) and import everywhere, OR document the rationale for keeping it copied so future agents stop reasoning about it. Small cost; high clarity gain.

**4. The `generateSongId()` for setlist IDs naming wart is now load-bearing**

Story 3.1 uses `generateSongId()` for `setlistId`. Story 3.4 reuses it for new setlists. Story 3.5 reuses it for both new songs and new setlists. Per AR-47 NanoIDs are general-purpose so functionally it's correct — but the name is now visibly wrong in 4+ places, and any future ID-type-by-name disambiguation becomes a wider search-and-replace.

**Fix for Epic 4:** Rename `generateSongId()` → `generateId()` and `web/src/lib/song-id.ts` → `web/src/lib/id.ts`. ~6 call sites. Trivial; do before Epic 4 adds more callers in Performance Mode entry / restore-from-storage.

**5. Story 3.3 iPhone bottom-sheet `<textarea>` deliberately did NOT reuse `InlineEditField`**

Story 3.3 Completion Notes: the sheet's commit action is the explicit `Done` button — not blur — so `InlineEditField`'s blur-commit semantics would have fired prematurely on focus loss. Correct call, but it surfaces a hidden assumption in `InlineEditField` (commit-on-blur is hard-coded). Right shape is probably a `commitOn: 'blur' | 'submit'` prop — but speculative until Epic 4 needs a second case.

**Lesson (not a failure):** Don't extract now — one duplication is fine. If Epic 4 adds a third sheet/modal/dialog that wants explicit-submit semantics, extract the prop then.

**6. Two TanStack Query placement patterns diverged within the same epic**

Story 3.4's iPhone `+ New setlist` affordance was placed inside `Home`, conditioned on `isIPhone()`. MacBook gets it via `TopNav rightActions`. No duplication of the link itself, but the placement logic is now split across two surfaces. Epic 4's Performance Mode entry will need similar split-surface placement. The pattern would be cleaner with a single "platform-conditional chrome slot" abstraction — but premature today with exactly two cases. **Wait for a third before extracting.**

## Key Insights

1. **Epic 2 retro's Critical Path discipline finally worked.** All 3 deferred prep items landed before Story 3.1. Lock this in.

2. **Design-note-first is the right pattern for novel features.** The `paste-to-parse-design.md` → Story 3.5 spec hand-off was the cleanest novel-feature delivery so far. Apply to Epic 4's Performance Mode entry behavior and wake-lock backoff policy.

3. **The commit checkpoint must NOT defer to orchestration without per-story granularity.** Either the dev-story commits, or the orchestration commits per-story with the story key in the commit message. The current "deferred — orchestration will handle it" boilerplate produces bundled commits that lose archaeology.

4. **Pattern-mirror stories (3.1, 3.2, 3.4) are very low-risk now.** When a story says "replicate songs.ts" or "replicate useSongMutation", the dev agent executes it almost mechanically. Risk concentrated in 3.5 (design-novel) and 3.6 (first stateful interaction).

5. **Zero new runtime deps across a 6-story epic is a real achievement.** AR-46 isn't decorative — the team genuinely considered DnD libraries for 3.6 and chose native, considered `natural` for 3.5 and implemented JW inline.

6. **The story-file Status vs sprint-status divergence is small now, will be load-bearing later.** Pick one as authoritative before Epic 4.

## Epic 4 Preview: Performance Mode

5 stories, all `backlog`. Dependencies on Epic 3 work — all satisfied:

- ✅ `Start performance ›` CTA stub mounted in 3.3 (`type="button"`, no handler) — Epic 4 wires `onClick`
- ✅ `CurrentlyPerformingStrip` slot marker in place in 3.3 — Epic 4 mounts the component
- ✅ `useTonightGig()` selector built in 3.2 — Epic 4 uses it for AR-25 pre-fetch
- ✅ Setlist overview fully renders sections + song rows in 3.3 — Performance Mode advances through these
- ✅ Outbox + flusher unchanged since 2.4 — Performance Mode read-only flow inherits stale-write suppression (AR-28) automatically

**Novel risk areas:**

1. **Story 4.1 — Single-tap navigation card layout.** First stateful navigation interaction on iPhone. Locked feedback memory: tap-anywhere collapses when content can scroll; separate nav into fixed chrome. Needs explicit attention — Performance Mode cards will likely have scrolling content.
2. **Story 4.2 — Wake-lock with persistent indicator and backoff.** Wake Lock API is platform-fragmented. iOS Safari requires standalone PWA context; behavior changes between iOS versions. **High-novelty — Sandy real-device proof required.**
3. **Story 4.4 — End-Performance state, last-song inert Next.** Intersects directly with locked feedback memory `[[feedback_no_terminate_on_advance_gesture]]`. AC must be explicit that Next goes inert/disabled, NOT transforms into "End performance."
4. **Story 4.5 — Backgrounding survives Tonight gig; pre-fetch Upcoming.** Touches AR-25 (pre-fetch) — never exercised yet. iOS PWA lifecycle is full of platform-specific edges. **High-novelty.**

**Suggested design-note-first candidates:**

- **Performance Mode entry/exit/resume state machine** — Story 4.3 is conceptually a state machine: idle → in-performance → backgrounded → resumed → exited. Lock the transitions in a design note before Story 4.1's spec is written.
- **Wake-lock backoff policy** — Story 4.2 needs explicit retry-and-backoff. Worth a paragraph pinning first-failure (silent retry), second-failure (indicator + backoff), user-recovery path.

**Epic 4 plan update required:** No. Epic 3 surfaced no findings that invalidate Epic 4's structure.
