## What Went Well

**1. Epic 1 retro Lesson #1 held across the whole epic**
Every user-visible story (2.1, 2.2, 2.5, 2.6) carried an explicit unchecked task checkbox for Sandy's manual proof. Stories 2.1 and 2.2 sat in `review` until Sandy ticked the iPhone install proof on 2026-06-16. Server-only stories (2.3, 2.4) correctly skipped the pattern. The discipline transferred cleanly from Epic 1's hardest lesson.

**2. The sync layer (Story 2.4) — the most complex single story to date — landed in one pass**
Outbox with per-record coalescing, IndexedDB-backed TanStack persister, flusher with exponential backoff, stale-write banner with iPhone+Performance-Mode suppression, error reporter wired to `window.onerror` / `unhandledrejection` / React `ErrorBoundary`, fetch wrapper with `x-server-now` drift check. 73 api tests + 146 web tests green; lint clean; build green. The architecture's pre-baked Decision 4 + outbox state machine + AR-20..AR-28 stack gave the dev agent an unusually clear blueprint, and the blueprint was correct.

**3. The hook-only consumption contract held (AR-45)**
Routes and components in Stories 2.5 and 2.6 never imported `sync/outbox.ts` directly. `useSongs()` / `useSong()` / `useSongMutation()` are the legal seams. The stale-write banner consumes a dedicated `stale-notice-store` via `useSyncExternalStore`. The boundary is real and is being respected — not just decorative.

**4. The iPhone manual proof surfaced a platform fact worth preserving**
Sandy's Story 2.2 test on 2026-06-16 revealed that iOS shares the cookie jar between Safari and the installed PWA at the same origin — `gigbuddy_session` carried across the install boundary. This was preserved in 2.2's dev notes, flagged in user memory, and explicitly tested for in Story 2.4's wiring. It's exactly the kind of "real device, real OS" finding that retrospectives should preserve and propagate.

**5. The schema-as-contract pattern from Epic 1 deepened**
Story 2.3 added three new shared Zod schemas (`SongSchema`, `SongPutInputSchema`, `ClientErrorReportSchema`) plus `ACTIVE_BAND_ID`. Both web and api consumed via `z.infer`. `web/src/lib/band.ts` was re-exported from `@gigbuddy/shared`. No parallel TypeScript types were defined anywhere. The discipline cost is zero once you're in the rhythm.

**6. `compareLww` was made generic on purpose, and that purpose paid off**
Story 2.3 deliberately wrote `compareLww<T extends { clientWrittenAt: string }>` so Story 3.1's Setlist routes can reuse it unchanged. This is forward-looking architecture work that the dev agent followed without prompting.

**7. Test growth was proportional and behavioral**
Story-by-story: 2.1 +6 (build-output assertions), 2.2 +12 (boot path + install gate), 2.3 +35 (server contract), 2.4 +111 (sync layer end-to-end), 2.5 +15 (Library), 2.6 +45 (Song Detail). No snapshot tests. The new sync-layer tests are real behavioral tests, not mock theater — they exercise outbox coalescing, retry/backoff state machine, stale-write banner suppression in Performance Mode, and the cache-invalidation path between optimistic write and Library re-render.

**8. Sandy's BMad cadence preference was respected**
After the Story 2.1 + 2.2 install proof checkpoint (necessary because subsequent stories depend on PWA install working), the epic ran through 2.3 → 2.6 without per-story manual proofs. One big iPhone+MacBook smoke is queued for epic close. This matches the locked feedback `[[feedback_bmad_epic_workflow_cadence]]` and worked well — no false stops.

## Challenges and Lessons Learned

**1. "Done in sprint-status" leaked from "code committed" — twice in one epic**

Story 2.3 was marked `done` in sprint-status.yaml while its files were still uncommitted; they ended up bundled with Story 2.4's commit (flagged in 2.4's review as a Decision item). Story 2.6 is at the same place now — marked done, but uncommitted at retro time (12 new files + 6 modified). The pattern is a real process gap.

**Root cause:** The dev-story workflow marks a story `done` based on tests/build/lint passing + checkboxes ticked. It does NOT verify that the working tree is clean and the changes are pushed. A story can pass every gate and still be sitting in the dev's working tree.

**Fix for Epic 3:** Treat the final step of the dev-story workflow as a verified commit. Either:
- (a) Make "all files committed and clean working tree" an explicit unchecked task checkbox in every story spec (analogous to Epic 1 Lesson #1 for manual deploys), OR
- (b) Add a pre-`done` shell guard to the dev-story workflow that fails if `git status --porcelain` is non-empty in the story's touched paths.

Option (a) is the lower-cost fix and matches the existing Epic 1 retro #1 pattern. Apply it from Story 3.1 onward.

**2. The 24-hour persisted-cache TTL on the iPhone PWA is a real production gap**

Story 2.4 attempted to set `maxAge: Infinity` on `createAsyncStoragePersister` — but that constructor doesn't accept that option, so the line was a silent no-op. Story 2.5's typecheck caught the dead field (TypeScript inference rejected it) and removed it without realizing the *intent* was load-bearing. The fix belongs one level higher: `maxAge: Infinity` on `PersistQueryClientProvider`'s `persistOptions` prop, where the persist-client library reads it.

**Impact:** iPhone PWA users (Sandy) who don't open the app for >24h will see an empty Library on next open until network connectivity restores. This is a regression against AR-20's "the cache survives a full page reload" intent — it survives 24h, not "forever".

**Lesson:** When a TypeScript error reveals a dead option, the dev agent should ask "did the original author *want* this behavior, just put it in the wrong place?" before deleting. In this case, removing the dead option was correct (the typecheck must pass), but the *intent* should have been carried forward as a deferred-work item with an explicit "fix belongs on `PersistQueryClientProvider` `persistOptions`" note. Story 2.5's dev agent did flag it for Sandy's awareness in Completion Notes, and it's now tracked in `deferred-work.md` line 89 — but the fix didn't actually land in Epic 2.

**Fix for Epic 3 prep:** One-line change to `web/src/sync/query-client.tsx` adding `maxAge: Infinity` to the `persistOptions` prop. Critical-path before Epic 3 because Setlists will live in the same cache.

**3. Vitest fake-timers + userEvent deadlock — twice**

Story 2.4's flusher tests and Story 2.6's Song Detail debounce tests both hit the same wall: `vi.useFakeTimers()` + `userEvent.setup({ advanceTimers: vi.advanceTimersByTime })` deadlocks when userEvent's internal awaits depend on the real microtask queue but timers are mocked. Both stories resolved the same way: switch to real timers + `waitFor`. Story 2.6 documented the resolution in AC-10 task checklist for future stories.

**Lesson:** The userEvent + fake-timers anti-pattern is now well-known in this codebase. Worth surfacing in a dev-environment reminders section of future story specs that include debounce or async-effect tests. (Not worth a `web/src/test-utils/` extraction yet — the fix is "use real timers + waitFor", which is shorter than any abstraction.)

**4. Architecture file-tree deviations need a clearer policy**

Story 2.4 made three judgment calls that diverged from the architecture's directory tree:
- `web/src/sync/lww.ts` not created (logic small enough to live inline in `flusher.ts`)
- `web/src/cache/persist.ts` not created (responsibilities split between `sync/persist.ts` and `sync/query-client.tsx`)
- `web/src/sync/query-client.tsx` (not `.ts`) because it exports a React component

All three deviations are *correct*. But the architecture tree (`_bmad-output/planning-artifacts/architecture.md`) still shows the old paths. Per CLAUDE.md: "Deviations require updating that document, not the implementation" — so the architecture should be updated to match reality, OR a "file-tree is illustrative, see story specs for ground truth" note added.

**Fix for Epic 3 prep:** Quick pass through architecture.md's directory tree section: either reconcile the three Story 2.4 deviations into the tree, or add a one-line note that the tree is illustrative.

**5. Epic 1 retro's parallel action items remained unfinished across all of Epic 2**

Two items from Epic 1 retro's "Parallel" bucket were not landed in Epic 2:
- `verbatimModuleSyntax` / `isolatedModules` contradiction in `tsconfig.base.json` (deferred-work.md, Story 1.1)
- Playwright API server startup for E2E (deferred-work.md, Story 1.1)

Neither caused a failure in Epic 2 (Stories 2.3–2.6 didn't add E2E tests; the tsconfig contradiction stayed silent). But:
- (b) is becoming load-bearing for Epic 3 — the Setlists home, paste-to-parse, and setlist creation flows are user-visible and benefit from real-backend E2E coverage that can't run today.
- (a) is a 1-line cleanup that has now waited 6+ stories.

**Decision (Sandy, 2026-06-18):** Land both before Epic 3. Add to critical-path prep below.

**6. Server-side TOCTOU and ISO-8601 edge cases were deferred**

Story 2.3 explicitly deferred several server-side robustness items: TOCTOU race in PUT (no DDB ConditionExpression), ISO-8601 timezone-format variations breaking lexicographic LWW, far-future `clientWrittenAt` poisoning, unbounded `listSongsByBand` scan, ZodError-on-DDB-drift surfacing as unstructured 500. All are tracked in `deferred-work.md` lines 61–71. All are acceptable for single-user V1.

**Lesson (not a failure):** These deferrals are appropriate and intentional, but `deferred-work.md` is becoming a real archive — 90 lines and growing. Worth a quick triage pass before Epic 5 (the ship-gate) to confirm which items are still acceptable at ship time and which need to land first. Not Epic 3 prep work, but flag for Epic 5.

## Key Insights

1. **The sync layer was the architectural risk of Epic 2, and it landed clean.** The pre-mortem outcomes documented in architecture.md (cached-401 strands user, mid-gig SW activation, iOS storage eviction) were each addressed by a specific Story 2.x design choice. The architecture earned its keep.

2. **Hooks-as-the-only-UI-seam is a discipline, not a constraint.** It cost zero in Epic 2 and bought a clean test surface where mutation behavior can be exercised without UI mounting.

3. **Manual proof checkboxes need to extend to "code committed" too.** The "marked done but not committed" pattern hit twice in one epic. Sprint-status should not be ahead of git.

4. **Real-device platform facts (cookie jar sharing, IndexedDB persistence) belong in user memory, not just dev notes.** This happened correctly for Epic 2; reinforce in Epic 3.

5. **The dev agent will silently delete options when typecheck complains, even when the intent was load-bearing.** The `maxAge` case is the canonical example. Worth a learned-lesson note in the dev-story workflow guidance.

## Epic 3 Preview: Setlists Home, Paste-to-Parse & Setlist Management

Epic 3 reuses Epic 2's patterns 1:1 for the server and sync layers, and introduces three novel features:

**Reused from Epic 2 unchanged:**
- `compareLww<T>` (generic by design in Story 2.3) — Story 3.1's Setlist PUT reuses it
- `api/src/ddb/` wrapper pattern — `api/src/ddb/setlists.ts` follows `songs.ts` shape
- `serverNowMiddleware` — already global
- Auth middleware — unchanged
- Whole-record PUT semantics (AR-23) — same contract for Setlists
- Outbox per-record coalescing — same rule, new `recordKey = "setlist:<bandId>:<setlistId>"`
- Mutation hook pattern — `useSetlistMutation` will mirror `useSongMutation`
- Stale-write banner — same Performance-Mode suppression
- Hooks-only consumption (AR-45) — unchanged

**Novel complexity in Epic 3:**
- **GSI1 query** (Story 3.1): first non-primary-key access pattern. `gsi1pk = BAND#<bandId>#SETLIST_BY_DATE`, date-ordered. Already in CDK from Story 1.3; just needs the query path.
- **Nested record shape** (Story 3.1): Setlist embeds `sections[]` containing `songs[]` (SongRef with `titleSnapshot` + optional `perGigAnnotation`). First time the schema has nested arrays.
- **Paste-to-parse** (Story 3.5): client-side text parsing into Matched / Fuzzy / Unknown song refs. **No precedent in Epic 2** — this is design-novel and warrants a quick design spike before the story spec is written (fuzzy-matching algorithm, section-name detection rules, Unknown→Match resolution UX).
- **Drag-reorder on MacBook** (Story 3.6): first stateful drag interaction. iPhone deliberately excluded (single-tap nav only per the locked design constraints).

**Dependencies on Epic 2:**
- Song CRUD + cache invalidation — Setlists embed `titleSnapshot` snapshots; renames don't propagate (AR-11)
- Library list pattern → Setlists home (Tonight / Upcoming / Past) is structurally similar but adds the temporal grouping
- Mutation hook pattern → Setlist mutation hook is a near-copy

**Epic 3 plan update required:** No. Epic 2 surfaced no findings that invalidate Epic 3's design.
