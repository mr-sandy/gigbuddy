## Deferred from: bootstrap execution (2026-06-15)

- **Function URL lock not enforced** (`infra/lib/stacks/api-stack.ts`, `web-stack.ts`) — CDK's `addFunctionUrl(authType: NONE)` auto-adds a public `lambda:InvokeFunctionUrl` permission that sits alongside the CloudFront-scoped permission in web-stack, rendering the lock ineffective. Direct Function URL hits return 200 instead of 403. Practical impact is low because the Hono auth middleware gates every route; the WAF rate-limit is the real exposure (bypassed on direct hits). Fix: replace `lambdaFn.addFunctionUrl()` with the L1 `CfnUrl` construct to suppress the auto-added permission, then only attach the CloudFront-scoped one. Also need to remove the `esbuild` root devDependency workaround by configuring NodejsFunction's `bundling.projectRoot` or `bundling.commandHooks` so CDK finds esbuild within the infra package.

## Deferred from: code review of 1-6-deploy-pipeline-with-two-stage-blackout-check (2026-06-12)

Not reproducible on `en-GB` + Node 22.
