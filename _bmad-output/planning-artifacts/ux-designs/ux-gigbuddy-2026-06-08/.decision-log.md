## Q2 — Home / landing screen per device *(2026-06-08, resolved)*

**Decision:** Home is gig-oriented on both devices. Same mental skeleton, device-tuned density. Library is a separate destination reached via header nav (MacBook) or a footer/inline link (iPhone), not the home.

### iPhone home

- Standard skeleton: `TONIGHT` (or next-gig) card, then `Recent` list. Same layout every day; on the day of a gig, the upcoming card carries a `TONIGHT` badge — layout doesn't change.
- Recents are real on iPhone — Sandy opens the phone between gigs *sometimes*, to check what was played last week or glance at a chord.
- No upcoming gig → empty state on the next-gig slot; recents still visible.

### MacBook home

- Setlists-first home: `TONIGHT` card, `Upcoming` list, `Recent gigs` list, stacked vertically.
- Top nav: `Setlists | Library` + a `New setlist` action.
- Library is a peer destination — not foregrounded on home but always one click away in chrome.

### Why this shape

- Sandy uses both devices; a radically different home on each adds mental tax for one person.
- Practice doesn't *always* start from a gig (occasional library-only sessions), but it usually does — gig-first home matches the brief's "setlist-driven library growth" principle.
- A calendar-as-home view doesn't earn its keep at 2–3 gigs/month.

### Options considered (rejected)

- **iPhone:** "Gig-day home collapses to just tonight" — too aggressive; recents are useful even on gig day.
- **MacBook:** "Same skeleton with inline library zone on the home" — too crowded; library deserves its own surface.
- **MacBook:** "Library-first dashboard" — practice mode is library-heavy, but the home is still a *gig* home; the library is a destination, not the entry point.

### Open within Q2

- Exact treatment of the `TONIGHT` badge (visual handling) — DESIGN.md call, not EXPERIENCE.md.
- "No upcoming gig" empty-state copy and shape — defer to State Patterns pass.

### Revision (during Q1)

Dropped Recents from iPhone home. Sandy's between-gigs use of iPhone is *forward-looking* (next gig, library browsing) — not *backward-looking* (last week). MacBook home keeps Recents (practice context often involves recalling what was prepped or played).

## Q1 — Setlist access pattern *(2026-06-08, resolved)*

**Decision:** Home itself is the primary access. The Setlists destination is one scrollable list, sectioned Tonight → Upcoming → Past. No separate "archive."

### iPhone

- Bottom tabs: `Setlists | Library`. Two tabs only (V1; band switcher folds in later — see Q5). Tabs are thumb-reachable, hidden in performance mode.
- `Setlists` tab content: `TONIGHT` (or next-gig) card at top, `Coming up` section, `Past` section. All in one scroll. No "All setlists →" link — the tab is the list.
- Tap a row → setlist overview (read for past, prep for upcoming).
- `Library` tab content: searchable / scrollable song list (Library specifics covered later).

### MacBook

- Header nav: `Setlists | Library` + a `New setlist` action.
- `Setlists` page: same sectioned shape (`Tonight` card → `Upcoming` → `Recent gigs / Past`). Vertical stack, more rows shown at once than iPhone.
- Library is a peer destination.

### Row format and behavior

- Row format: `Venue · Date` (mirrors how Sandy already titles real-example gigs: "Howlin Wolf - 2nd May").
- One tap, one destination. No "view vs open" split. Past rows land on setlist overview in a read-leaning state; upcoming rows land in prep-leaning state. Same surface, different defaults — handled in setlist-overview spec, not access.
- No "Repeat this setlist" affordance. Implies setlist templating; V2 question.
- No delete-from-list affordance. Delete lives on the overview itself if it ever earns a place — likely not V1.

### Why

- At 2–3 gigs/month, the volume doesn't justify search/filter or year-grouping in V1. Scroll covers years.
- One destination per gig keeps the model simple — no separate "preview" vs "performance-ready" entry points.
- iPhone tabs hidden in performance mode preserves the "performance mode is sacred" principle.

### Options considered (rejected)

- **Calendar view as primary access** — empty cells dominate at this volume.
- **Search-first archive** — premature for V1 volume.
- **Year-grouped archive** — premature; would matter in year three.
- **Separate "Archive" screen for Past gigs** — adds chrome without saving anything; one scroll is shorter than one tap-then-scroll.
- **Tabs for Upcoming / Past within Setlists** — hides Past behind a switch; flat scroll is friendlier and matches the MacBook split-with-divider preview Sandy already approved.

### Open within Q1

- Bottom-tab labels — `Setlists` / `Library` are working titles; could revisit at copy pass.
- Whether `New setlist` is accessible from iPhone or MacBook-only. Brief implies prep happens on MacBook ("paste WhatsApp text"); deferred to Setlist Creation spec.

## Q3 — Practice ↔ performance mode transition *(2026-06-08, resolved)*

**Decision:** Performance mode is an iPhone-only state, entered by an explicit primary CTA on the iPhone setlist overview. Exit returns to overview with state preserved.

### Entry

- iPhone setlist overview has a primary CTA: `[ Start performance › ]`.
- Tap fires immediately into the performance card view, song 1 of the setlist (first non-empty section — typically 1st Set).
- **No confirm step.** The CTA is the deliberation. An extra sheet would slow the moment between the count-in and the first downbeat.
- The CTA is visually unmistakable (primary, large, distinctive) so accidental fire is implausible.

### Path from home

- iPhone Tonight card → tap → setlist overview (prep view).
- Setlist overview → tap `Start performance` → performance card view.
- **Two taps from home to enter performance. No smart routing** (no "after 7pm goes straight to performance"). One rule, predictable.

### Exit / overview peek

- Performance mode has a small `×` (or chevron) tucked in a corner (top-left, inside safe-area inset, low-prominence but reachable).
- Tap → returns to setlist overview.
- **State persists.** Overview shows a "Currently performing: [song]" strip with `[ Resume › ]`. Wake lock stays active. Position remembered.
- Performance state ends only when Sandy navigates away from the setlist entirely (Library tab, another setlist, app fully closed). At that point: release the wake lock; clear "current song."

### Sticky performance state

- If the app backgrounds (iPhone locks, home button accidentally hit, notification overlay) and is re-opened, the user lands back on the **performance card view for the current song** — not on home. The performance state is sticky across app lifecycle until explicitly ended.
- Implementation concern, but flagged here so the UX spec can rely on it.

### MacBook side

- MacBook does **not** have performance mode. There is no `Start performance` CTA on MacBook surfaces. (If a future case appears — e.g., a MacBook in a rehearsal-room context — revisit; not V1.)

### Why

- Explicit, single-trigger entry honors "performance mode is sacred."
- Sticky-state exit means "peek at overview" is safe — you cannot lose your place by tapping the wrong thing.
- No long-press, no precise gestures — honors the sweaty-hands constraint.

### Options considered (rejected)

- Confirm sheet on entry — adds friction at the wrong moment.
- Long-press entry — precise gesture, dim light, sweaty hands.
- Swipe-down to reveal chrome — less discoverable than a visible affordance.
- Tap song title to peek at overview — title is large and central; risk of accidental hit.
- Direct entry to performance from Tonight card on home — too aggressive; setlist overview is the natural staging surface where pre-gig per-gig annotations get added.

### Open within Q3

- Exact visual treatment of the `×` exit and the "Currently performing" strip — DESIGN.md call.
- Whether `Start performance` is also a keyboard-shortcut on MacBook (no — out of scope for V1; MacBook has no performance mode).
- What happens if Sandy hits `Start performance` on a setlist with zero songs in the 1st Set (only Reserve populated). Edge case; defer to State Patterns.

## Q5 — Band switching in V1 chrome *(2026-06-08, resolved)*

**Decision:** V1 shows the band as passive context on MacBook only. No switcher. No band-management UI. Data model remains multi-band.

### Specifics

- MacBook header: `GigBuddy · The Jack Ruby 5` — informational label; not interactive in V1.
- iPhone: no band indicator anywhere in V1 chrome.
- No "create band" / "manage bands" screen in V1. Bands are seeded; only Jack Ruby 5 carries content.
- V2 hook: when a second band gets populated, the MacBook label becomes interactive (dropdown), and iPhone gains a switch affordance (location TBD in V2 design).

### Why

- A switcher with one option implies a choice the user doesn't have.
- Showing band as context (MacBook) is honest about the data shape without pretending there's a switch.
- iPhone screen is precious; the band context is implicit (Sandy knows which band he's prepping for).

### Options considered (rejected)

- Active switcher on both devices even with one band — switch-to-nowhere ergonomics, dead weight in V1.
- No band indicator anywhere — too clean; loses honest signal of the data shape.
- Band management UI in V1 — premature; no V1 user need.

### Open within Q5

- V2 switch location on iPhone — defer.
- Exact visual treatment of the MacBook header band label — DESIGN.md call.

## Q8 — Edit surface (when and where) *(2026-06-08, resolved)*

**Decision:** Two distinct edit surfaces, both inline, both silent auto-save. Per-gig annotations are free-text in V1; structure extracted in V2 once recurrence patterns are visible.

### Surface A — Permanent song record (Library → song detail)

- No edit mode. View IS edit. All fields are inline-editable.
- Field set (V1): title, key, patch/sound, chord chart (monospaced text block), performance notes, practice notes (free text + embedded links).
- Click into a field → type → click away. Auto-save on blur. Debounce on rapid input.
- No "Edit" button. No "Save" button. No modal.

### Surface B — Per-gig annotation (Setlist overview → song row)

- Setlist overview shows each song as a row; section headings (`1st Set` / `2nd Set` / `Reserve`) preserved.
- Default row carries section, song title, and any existing per-gig annotation rendered visually distinct from the permanent record.
- Tap a row → expands inline (MacBook) / opens sheet (iPhone) with one text field: **"Notes for this gig"** + a small preview of the permanent record (read-only mirror, for reference).
- Annotation is free-text in V1. Structure (soloists, staging, context) is a V2 question once the data shows what recurs.
- Auto-save on blur; same save behavior as Surface A.
- Annotation is never written back to the permanent song record. Layered, not edited.

### Save behavior (both surfaces)

- **Silent optimistic save.** No "Saving..." or "Saved." indicator visible by default.
- Sandy is sole user; no concurrent-edit conflicts; loss risk is low; this is a personal tool.
- Error toast appears **only** if save actually fails (network drop, server unreachable).
- No "leaving page during in-flight save" warning. Queue and trust.

### Performance-mode rendering of annotations

- Per-gig annotations show in performance mode card view, **visually distinct** (different weight / color / accent — DESIGN.md call) from the permanent record. Sandy can spot the "vocal tonight?" cue at a glance.

### What this rules out (V1)

- Dedicated edit-mode toggle / Edit button / Save button.
- Modal forms for editing.
- Version history, undo across sessions, "Editing as Sandy" indicators.
- Structured per-gig field schema (V2).

### Why

- Friction-free editing matches current Apple Notes habit; the brief's whole premise is replacing that workflow without making it heavier.
- No concurrent users → no need for save ceremonies.
- Two-surface model honors the brief's invariant: per-gig annotations are layered, never editing the permanent record.
- Free-text annotation now defers structure to evidence — structure built before data is structure built wrong.

### Open within Q8

- Exact visual treatment of "annotation distinct from permanent record" — DESIGN.md call.
- Field validation: do we constrain `key` field to known keys? Almost certainly not — Sandy uses shorthand like `Cm/F`, `Fm Blues`. Free text wins.
- Single-level undo behavior — browser-native should suffice; flagged if testing reveals gaps.

## Q4 — In-performance navigation between songs *(2026-06-08, resolved)*

**Decision:** Fixed nav chrome at top and bottom; chord-chart and annotations scroll in the middle. `NEXT ›` is a large, right-biased primary button. `‹` back is small. No tap-anywhere; no split tap zones.

### Card layout

- **Top chrome (fixed):** `×` exit (top-left, small, safe-area inset), position indicator `3 / 14` (top-right, small).
- **Title row (fixed):** Song title (huge, primary). Below it: key + patch (`Gm · Jazz Organ P14`).
- **Scroll region (middle):** chord chart + per-gig annotation. Scrolls vertically when content overflows.
- **Bottom toolbar (fixed):** `‹` back (left, small), big `NEXT ›` button (right-biased, ~half-width × ≥ 80pt tall), with small next-song title preview ("Sunny") rendered under the button.

### Why fixed chrome

- Tap-anywhere collapses when content can scroll — finger-down is the start of both gestures, and the OS distinction (move-before-lift) is unreliable under sweaty-hands / divided-attention conditions. See `feedback_check_scroll_conflict.md`.
- Real-content survey: ~70–80% of Jack Ruby 5 songs fit one screen at the type sizes we want; ~20–30% (SUNNY with guitar-solo alternate keys, SUMMER FIELDS OF JOY with theme + appendix) need scroll. The design has to handle both shapes with one model.
- Fixed nav also keeps title + key + patch visible while scrolling the chart — those are the orientation cues Sandy needs at any moment.

### Asymmetry: NEXT is dominant

- Forward is ~95% of moves. Back is a correction.
- Visual weight matches: `NEXT ›` is huge and right-biased; `‹` back is small.
- Easy to hit at arm's length without precision.

### Next-song preview

- Small subtext under `NEXT ›` showing the next song's title (e.g., "Sunny").
- Lets Sandy grab the right patch *before* tapping advance.

### End of setlist

- On the last song, `NEXT ›` does **not** loop. It either becomes inert or transforms into `End performance ›` (decision deferred — State Patterns).

### Mid-setlist skipping

- Exit (`×`) → setlist overview → tap target song row → drops into performance card for that song. Three taps. Reuses existing routing from Q3.

### Spatial separation of controls

- `×` exit: top-left.
- `‹` back: bottom-left.
- `NEXT ›` advance: bottom-right (dominant).
- Position indicator: top-right (informational only).
- No two destructive/state-changing controls share a corner.

### Trade-off

- Fixed chrome eats ~140pt total (60pt top + 80pt bottom + safe-area insets). Leaves ~80% of usable height for title/meta/chord chart on iPhone 13. Still plenty for short songs; long ones scroll.

### Options considered (rejected)

- **Tap-anywhere to advance** — collapses with vertical scroll.
- **Split tap zones (left = back / right = forward)** — same scroll conflict + requires holding a mental model under stress.
- **Horizontal swipe** — fragile with sweaty hands and a phone resting on the Nord.
- **Auto-fit chord chart (no scroll)** — would force tiny text on long songs; violates accessibility floor.
- **Within-song pagination (page 1 of 2)** — conditional tap behavior is exactly the unpredictability the brief forbids.

### Open within Q4

- "End of setlist" treatment — `NEXT ›` inert vs. transforms vs. exits performance. Defer to State Patterns.
- Exact visual treatment of all chrome — DESIGN.md call.
- Whether the next-song preview is read aloud by VoiceOver or skipped. Accessibility pass.

## Q6 — Portrait vs landscape *(2026-06-08, resolved)*

**Decision:** Portrait-only for performance mode. Locked. Device rotation is ignored.

### Why

- iPhone 13 rests naturally portrait on the Nord.
- Chord notation reads line-by-line; vertical stacking matches reading flow.
- Fixed top/bottom chrome from Q4 lives more comfortably in portrait.
- One layout, one rule.

### Wide-chord concern (deferred to DESIGN.md / typography)

- Widest real-example chord line: `F#m A7 Dmaj7 G#m7 C#7` (~25 chars). At 32pt monospaced this exceeds the 390pt iPhone 13 portrait width.
- Resolution options (DESIGN.md call, not orientation): slight font-size step-down for long lines, wrap with a hanging indent, or tighter monospace metric.

## Q7 — Chord chart density + sparse state *(2026-06-08, resolved)*

**Decision:** Chord chart is monospaced free-text with **light visual parsing** for conventions Sandy already uses. Sparse-content state is **honest empty space** — no placeholder.

### Reframe — "4-line blocks" was a Claude Design heuristic

Real chord charts vary from 0 lines (FOOTPRINTS) to 10+ lines (SUNNY's main + `{for guitar solo}` alternate key). The right framing is: monospaced, generously spaced, **variable-length**, scrolls when long. Length is whatever Sandy authored. (Scroll is already handled by Q4.)

### Light visual parsing — what's parsed

- **`{...}`-wrapped lines** become visual section breaks: a thin horizontal rule with the inline label centered ("──── for guitar solo ────"), smaller weight than chord lines, possibly italic.
- **Blank lines** in the source preserved as vertical breathing space.
- **URLs in practice notes** (practice mode only) are tappable. In performance mode, URLs are not present in the chord field anyway.
- **No chord-symbol parsing.** No coloring of chord tokens. No diagrams. Sandy's `Gm blues`, `descending line F, E, Eb, D`, and shorthand like `Bb (top two inversion of standard blues lick)` are left as authored text.

### What this does NOT parse

- Title content — rendered as authored. `SUNNY - Bobby Hebb - SOLO` shows as one heading; the app does not split `SOLO` out into a badge or `Bobby Hebb` into an artist field.
- Key/patch lines — rendered as authored. `F#m(Ian) Fm(Sandy)` and `Wurlitzer (P13)` show as typed.
- Transpose flags — rendered as authored. `(P33 IS TRANSPOSED!)` and `TRANSPOSE?? -2 = Cm/F` show as typed.
- All these match Q8 (free-text now, structure later).

### Sparse-content state (FOOTPRINTS-shaped songs)

- Title + key/patch render normally.
- Chord-chart area is **blank**. No placeholder, no "— no chart —" caption, no "Add chart" CTA.
- Card height stays full; the empty space below the title is acceptable — the bottom toolbar still anchors the screen.
- Honest about what's there. Sandy knows FOOTPRINTS; the app doesn't need to apologize for the absence of notes.

### Why

- Light parsing honors a convention Sandy already authored without imposing new structure.
- Free text rule (Q8) survives — parsing is purely *typographic* response to shape, not content transformation.
- Heavy parsing would over-structure loose notation and fight Sandy's shorthand.
- Honest sparse state matches the brief's "context-appropriate information" principle: when a song needs no notes, the card needs no clutter.

### Options considered (rejected)

- **Pure free-text monospaced** — `{for guitar solo}` renders as literal text; cluttered visually, less skimmable in dim light.
- **Heavier parsing (chord recognition, color-coding, diagrams)** — premature; loose notation resists structure.
- **Sparse placeholder text** — adds chrome where none is needed; the empty space *is* the design.
- **Tightened card on sparse content** — different card shape on sparse vs dense violates "predictable under stress."

### Open within Q7

- Exact visual treatment of `{...}` section breaks (rule weight, label color, italic vs caps, spacing) — DESIGN.md call.
- Whether per-gig annotations render *inside* the chord area or *outside* (above the chord chart? below the title?) — defer to per-feature spec pass.

## Feature inventory + V1/V2 cuts *(2026-06-08, resolved)*

### V1 active features

1. **Band scope (passive).** One active band (Jack Ruby 5). MacBook header shows band label; iPhone shows nothing. Multi-band data model preserved for V2.
2. **Song Library** (band-scoped CRUD). Fields: title, key, patch/sound, chord chart, performance notes, practice notes (with embedded URLs). Inline edit, silent auto-save. Library list view + song detail view.
3. **Setlist Management.**
   - **Paste-to-parse creation:** paste WhatsApp text → parse titles + section structure → fuzzy-match against library → flag unknowns → gap-fill → save.
   - **Manual creation:** add songs one at a time from a setlist edit view.
   - **Edit:** add/remove songs, reorder within section (drag-and-drop, MacBook), move between sections, edit per-gig annotations.
   - **View:** setlist overview by section, per-gig annotations visible, `Start performance ›` CTA on iPhone.
4. **Performance Mode** (iPhone only). Single-song card view per Q4 + Q7. Wake-lock. Sticky state across backgrounding. Exit → overview with `Resume ›`.
5. **Practice Mode** (MacBook primary). Library browse + edit; setlist creation + edit; tappable URLs in practice notes.
6. **Navigation chrome.** iPhone bottom tabs (Setlists / Library), hidden in performance mode. MacBook top nav (Setlists / Library / `New setlist`).

### Explicitly NOT V1

- Setlist analytics, repertoire balance reporting, "songs not played in 3 months" alerts.
- Bandmate access / sharing.
- Apple Notes import (manual or paste only).
- Audio / MIDI / hardware integration.
- Multi-band UI surfacing (passive context only).
- Setlist intelligence / suggestions.
- Setlist "ready to perform" state machine (V2; every setlist is performable in V1).

### Setlist sections

- **Variable count and naming.** Each setlist has 1+ sections. Sometimes 1 set, sometimes 3. Reserve is optional, not built-in.
- Default first section on a new setlist: `Set 1` (renameable).
- Section names are free text (typical values: `Set 1`, `Set 2`, `Set 3`, `Reserve`, `Encore`, but any string is valid).
- Paste-to-parse infers section boundaries from the pasted text (`----` rule + heading line).

## Real-content edge cases *(2026-06-08, resolved)*

Most edge cases resolved by earlier decisions (Q7 free-text rendering, Q8 free-text annotations). Two surfaced here and resolved.

### Capitalisation is NOT a signal *(correction landed 2026-06-08)*

Earlier annotation in `real-example.md` claimed `ALL CAPS = confirmed` / `mixed case = TBA`. Sandy corrected: the all-caps was manual visual styling in his Apple Notes; lowercase titles were forgot-to-uppercase oversights. There is no semantic content in case.

**Implications for the data model and UX:**
- Library has a single canonical title per song. Sandy edits it on the song detail page.
- Setlists reference songs by record. The setlist view renders the **canonical library title**.
- **No per-gig `displayTitle` field. No verbatim casing preserved.**
- TBA / optional songs are expressed as a **section** (e.g., a `Reserve` section), not as a per-song flag. This is why sections are variable (sometimes Reserve, sometimes not).
- Paste-to-parse normalises case + whitespace for matching but stores only the song reference; the canonical title is what gets displayed everywhere downstream.

**Source docs amended:** `_bmad-output/planning-artifacts/ux/design-handoff/real-example.md` and `real-example-2.md` updated to remove the false-signal claim.

### Paste-to-parse: three-state model

| State | Visual | Behavior |
|---|---|---|
| `✓ Matched` | Quiet — green check or similar | Title matches an existing library song (case + whitespace normalised). No user action. |
| `? Did you mean …?` | Yellow / attention | Fuzzy candidate found (Levenshtein distance under threshold, or matches a similar normalised form). One-tap accept; one-tap "no, new song." |
| `+ Unknown` | Red / attention | No match candidate. Inline action: create a minimal song record now (just the title; fields filled later in library). |

- Gap-fill happens **inline** on the parse-review screen. Not a separate workflow step. You see the parsed result, resolve yellows and reds, then save.
- Real-example evidence requiring this: `KELVINGROVESTREET` vs `KELVINGROVE STREET` (same song, must fuzzy-match).

### Other content edge cases — already resolved upstream

- **Multi-key songs (`F#m(Ian) Fm(Sandy)`):** Q7 — free-text key field, rendered as authored.
- **Transpose state (`P33 IS TRANSPOSED!`, `TRANSPOSE?? -2 = Cm/F`):** Q7 — free-text in patch/notes, rendered as authored.
- **Per-gig soloist annotations (`WATERMELON MAN – Ivan Ian John`):** Q8 — free-text per-gig "Notes for this gig" field.
- **Inline staging notes (`[GUITAR CHANGE]`):** Q8 — free-text per-gig annotation.
- **Section headings within chord chart (`{for guitar solo}`):** Q7 — light visual parsing renders as a visual section break.
- **Minimal entries (FOOTPRINTS = title + "Piano"):** Q7 — sparse state, honest empty space.

### Open within edge cases

- Whether the "Did you mean?" affordance offers multiple candidates (top-3) or only the highest-scoring one — defer to per-feature spec pass.
- Levenshtein threshold for fuzzy-match — implementation detail, not UX.
- What happens when the same parsed title matches two existing library songs equally — defer to State Patterns.

## Per-feature spec passes *(2026-06-09, resolved)*

### Setlist overview

**Shape (both devices):**
- Header: venue · date · time — compact, top of screen.
- Optional gig-level notes below header.
- Body: sections with section headings + song rows; per-gig annotation rendered as a small subline under each row, visually distinct from canonical song notes (per Q7 / Q8).
- Row format: section heading, song title (canonical from library), optional per-gig annotation subline.

**iPhone-specific:**
- Bottom-fixed `Start performance ›` action bar, **above** the Setlists/Library tab bar. Always reachable without scrolling.
- Tap a row → inline expand (or sheet) for editing per-gig annotation (Q8 Surface B).
